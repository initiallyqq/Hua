# 花箴 · 产品需求文档

## 一、产品定义

### 1.1 一句话定义

花箴是一个**面向写作过程的、有人情味的桌面写作伴侣**。

### 1.2 产品哲学

- **关注过程，而非产出** —— 语雀关心文档和知识库，花箴关心"你是怎么写出来的"
- **工具是陪伴，而非命令** —— AI 不是助手，是同桌
- **每一次写作都是一次旅程** —— 可以被回放、被看见、被理解

### 1.3 用户画像

- 写作者（日记、随笔、技术文章）
- 注重创作体验和灵感记录的人
- 对"数据≠隐私"敏感、偏好本地工具的人
- 追求审美和情绪价值的人

## 二、三大产品方向

### 方向 A：墨迹回放（Ink Playback）

**一句话：像视频一样回放你的写作过程，AI 帮你看见自己是怎么写的。**

#### A.1 核心功能

| 功能 | 描述 | 优先级 |
|---|---|---|
| A1.1 操作录制 | 每次按键、粘贴、删除、光标跳转，带时间戳 | P0 |
| A1.2 回放引擎 | 按时间轴播放文本变化，可播放/暂停/拖动进度条 | P0 |
| A1.3 速度控制 | 1x / 2x / 4x / 8x / 16x 加速播放 | P1 |
| A1.4 AI 标记节点 | 自动标注纠结点、停顿点、流畅点、跳跃点 | P1 |
| A1.5 节点跳转 | 点击标记直接跳到对应时间点 | P1 |
| A1.6 写作画像 | 基于多篇笔记数据生成写作习惯报告 | P2 |
| A1.7 分享导出 | 将回放导出为视频或分享链接 | P2 |

#### A.2 数据模型

```
InkSession {
  noteId: string
  startTime: timestamp
  endTime: timestamp | null
  
  events: InkEvent[]
    - type: "insert" | "delete" | "select" | "cursor" | "paste"
    - timestamp: number (ms since session start)
    - position: number (cursor offset)
    - length?: number (selection length)
    - text?: string
    - beforeContent: string (optional, for undo)
}

InkNode {
  timestamp: number
  type: "hesitation" | "pause" | "flow" | "jump"
  duration: number
  startEventIndex: number
  endEventIndex: number
  aiComment: string  // AI 生成的注解
}
```

#### A.3 用户故事

1. 写完一篇笔记后，打开"墨迹回放"，看到 AI 标出的 3 个纠结点，点击跳转到对应位置
2. 打开一篇旧笔记，看到当时写作的完整过程
3. 查看"写作画像"，发现自己在深夜写作效率更高、凌晨容易反复删改

#### A.4 存储方案

- 录制数据存本地的时序文件（如 SQLite 或 `.ink` 二进制格式）
- 不随 `.md` 笔记上传或迁移
- 保留策略：设定最大保留天数或单篇最大事件数

### 方向 B：共笔模式（Co-Write）

**一句话：人和 AI 像同桌一样在同一张纸上轮流写。**

#### B.1 核心功能

| 功能 | 描述 | 优先级 |
|---|---|---|
| B1.1 共笔会话 | 创建一个独立于原始笔记的共笔空间 | P0 |
| B1.2 AI 身份预设 | 续写者/追问者/反对者/诗意者/自定义 | P0 |
| B1.3 交替写作 | 写完一段 → 按回车或快捷键 → AI 接下一段 | P1 |
| B1.4 自由穿插 | 双光标模式，人和 AI 可在任意位置写 | P2 |
| B1.5 角色共笔 | 设定双方身份角色，针对场景共写 | P2 |
| B1.6 作者标记 | 正文中区分"人写"和"AI 写"的部分 | P1 |
| B1.7 合并到笔记 | 共笔完成后将内容有选择地合并回主笔记 | P1 |

#### B.2 会话流程

```
[用户选择"共笔模式"]
  → 新建共笔会话（独立于当前笔记）
  → 选择 AI 身份：续写者/追问者/反对者/诗意者
  → 开始共笔

[共笔循环]
  1. 用户写一段（墨色/黑色文字）
  2. 用户按 Enter / 点"轮到 AI"
  3. AI 生成下一段（bamboo 绿色文字）
  4. AI 写完，用户可继续写（可改 AI 的，也可留）
  5. 重复 1-4

[结束共笔]
  → 保存会话（独立）
  → 可选择"合并到笔记"（手动选择保留哪些段落）
```

#### B.3 Prompt 设计

```
你是一个共笔搭档。你正在和一位写作者共用一张纸。
你的身份是：[续写者/追问者/反对者/诗意者]

写作规则：
1. 你只能写一段，一段 1-3 句话。
2. 写出对方意料之外但情理之中的内容。
3. 不要评价对方的文字，也不要回答问题——你是来一起写的，不是来对话的。
4. 保持和原文一致的语言和语调。
5. 一段写完后停笔，让对方继续。

当前全文（<human> 是人写的，<ai> 是你之前写的，交替标注）：
{full_text_with_markers}

轮到你了，写下一段：
```

#### B.4 用户故事

1. 想写一篇散文但不知道如何开头，开启共笔选"追问者"，AI 不断追问帮你挖素材
2. 写文章总觉得自己写得太严肃了，开"诗意者"模式让 AI 打破自己的写作惯性
3. 一个想法反复推敲不下去，开"反对者"模式让 AI 帮你找反例

### 方向 C：写作花园（Writing Garden）

**一句话：AI 不说话，用一株植物的生长状态反映你的写作习惯。**

#### C.1 核心功能

| 功能 | 描述 | 优先级 |
|---|---|---|
| C1.1 桌面植物 | 系统托盘或桌面上显示一株生长的植物 | P0 |
| C1.2 写作灌溉 | 写多少字 = 浇多少水，植物生长 | P1 |
| C1.3 连续写作 | 连续天数越长，植物开花越盛 | P1 |
| C1.4 状态反馈 | 断更 = 叶片枯萎，持续高产 = 开花 | P2 |
| C1.5 主题植物 | 不同植物代表不同写作目标 | P2 |
| C1.6 花园全景 | 打开应用可看到全部植物的状态总览 | P2 |
| C1.7 季节系统 | 现实季节影响植物的外观 | P3 |

#### C.2 数据模型

```
Plant {
  id: string
  species: "梅花" | "竹子" | "兰花" | "菊花" | "睡莲"
  birthDate: timestamp
  growthStage: 0-100  // 生长阶段
  health: 0-100       // 健康值
  lastWaterDate: timestamp
  streakDays: number  // 连续写作天数
  bloomed: boolean    // 是否开花
  stats: {
    totalChars: number
    totalSessions: number
    longestStreak: number
  }
}
```

#### C.3 视觉设计原则

- 东方水墨风 + 现代 Minimal
- 低饱和度，和花箴现有审美一致
- 植物动画流畅、克制（不用弹窗或彩带）
- 状态变化是缓慢渐变，不是突然翻转

#### C.4 用户故事

1. 用户连续写了一个星期日记，每天早上打开花箴，桌面的竹子长出了新叶子
2. 当"健康值"下降到危险线但用户超过了预计写作时间还没打开应用，植物会微微低垂，无声提醒
3. 用户完成了第一个"连续写作 30 天"里程碑，植物开出第一朵花

#### C.5 注意点

- C 方向的数据依赖 A 方向（墨迹数据提供输入）
- C 方向是长期陪伴功能，初期可只做基础植物 + 简单生长逻辑
- 不要做复杂游戏化（成就、等级、排行榜）
- 植物可以钉在桌面（Tile 窗口），用户随时看到

## 三、方向关系图

```
┌─────────────────────────────────────────────────────────┐
│                   墨迹回放 (A)                           │
│  录制每一次按键 → 回放 + AI 标记 → 写作画像             │
└───────────────┬─────────────────────────────────────────┘
                │ 提供数据
                ▼
┌─────────────────────────────────────────────────────────┐
│                   写作花园 (C)                           │
│  写作数据 → 植物生长 → 无声陪伴                         │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                   共笔模式 (B)                           │
│  独立会话 → AI 身份 → 交替写 → 合并到笔记               │
└─────────────────────────────────────────────────────────┘
```

- A（录数据）和 B（共笔）并行开发
- C 依赖 A 的墨迹数据积累，建议后期启动

## 五、技术架构要点

### 前端

- **墨迹录制**：主窗口 `MainWindow.tsx` 中现有的 `onChange` 增加事件记录，与内容存储分离
- **回放 UI**：可选方案为基于现有 textarea 的"模拟输入"或基于 CodeMirror 的虚拟渲染
- **共笔 UI**：利用已有的 `textarea` 增加 AI 文字渲染层（不同颜色），或划出独立共笔面板

### 后端（Rust）

- **墨迹存储**：在 `NoteStore` 同级新增 `InkStore`，数据存本地 SQLite（实现简单）或独立 `.ink` 文件
- **共笔 session**：`NoteStore` 扩展 `CoWriteSession` 模型，存为独立文件

### AI

- **墨迹分析**：批量事件数据压缩后调 DeepSeek API 分析节点
- **共笔续写**：分离 DeepSeekChat 的对话逻辑，封装 `coWrite(session, identity) → string` 接口

## 六、竞品差异化

| 竞品 | 花箴差异 |
|---|---|
| Obsidian | 关注知识管理 → 花箴关注**写作过程** |
| Notion | 团队协作+文档 → 花箴是**个人陪伴** |
| 语雀 | 云端知识库 → 花箴是**本地优先、隐私安全** |
| VS Code | 编辑器+插件 → 花箴是**完整的体验设计** |
| 微信读书笔记 | 阅读+笔记 → 花箴是**写作者的私人空间** |
