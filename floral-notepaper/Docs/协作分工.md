# 花箴 · 多人协作分工方案

## 一、方向分工建议

三个方向可以并行推进，互不阻塞：

### 方向 A：墨迹回放

**谁适合**：有前端经验 + 数据结构设计能力的人

| 人 | 负责模块 | 说明 |
|---|---|---|
| 前端 1 | 操作录制（事件采集） | 在 MainWindow.tsx 的 onChange 中插入录制逻辑 |
| 前端 2 | 回放播放器 UI | 进度条、播放/暂停、速度控制、时间轴 |
| 后端 1 | 墨迹存储（Rust 端） | InkStore 数据模型 + 本地持久化 + 读取接口 |
| AI 工程师 | 墨迹分析 | 事件序列压缩后调 DeepSeek，识别停顿/纠结点 |

**接口约定**：

```
frontend → rust: saveInkEvents(noteId, events: InkEvent[])
frontend ← rust: loadInkSession(noteId): InkSession
frontend ← rust: loadInkNodes(noteId): InkNode[] (AI 分析后的标记)
```

---

### 方向 B：共笔模式

**谁适合**：有前端交互设计能力 + Prompt 工程经验的人

| 人 | 负责模块 | 说明 |
|---|---|---|
| 前端 1 | 共笔会话 UI | 新建会话、身份选择、交替写界面 |
| 前端 2 | 作者标记渲染 | human/AI 文字不同颜色，区分展示 |
| 后端 1 | 共笔会话存储 | CoWriteSession 模型 + CRUD |
| AI 工程师 | Prompt 设计与调优 | 不同身份预设的 Prompt + DeepSeek 响应适配 |

**接口约定**：

```
frontend → rust: createCoWriteSession(noteId, identity): SessionId
frontend → rust: appendHumanText(sessionId, text)
frontend → rust: requestAITurn(sessionId): string (AI 返回的下一段)
frontend → rust: getSession(sessionId): CoWriteSession
frontend → rust: mergeToNote(sessionId, selectedRanges)
```

---

### 方向 C：写作花园

**谁适合**：有前端动画/Canvas/SVG 能力的人

| 人 | 负责模块 | 说明 |
|---|---|---|
| 前端 1 | 植物渲染 | Canvas/SVG 植物生长动画、多状态 |
| 前端 2 | Tile 窗口集成 | 植物以 Tile 窗口钉在桌面 |
| 后端 1 | 花园数据 | Plant 数据模型 + 生长算法 + 依赖墨迹数据 |

**注意**：C 方向依赖 A 方向的墨迹数据，建议在 A 的录制模块完成后启动。

## 二、并行开发时序

```
Week 1-2:     A操作录制 ─┐                   B共笔会话UI ─┐
                          ├ 并行              ├ 并行
               A存储模块 ─┘                   B后端存储 ───┘

Week 3-4:     A回放播放器 ─┐                 B交替逻辑+Prompt ─┐
                          ├ 并行              ├ 并行
               A AI标记 ───┘                 B合并到笔记 ──────┘

Week 5-6:     A写作画像     │     C花园基础 ─── (开始依赖A的数据)
              集成测试 ──────┴────── 正式发布 Phase 1
```

---

## 三、推荐人员配置（最小值）

| 角色 | 人数 | 方向 |
|---|---|---|
| 前端工程师 | 2 人 | A 录制+回放 / B 共笔 UI |
| Rust 后端工程师 | 1 人 | A 存储+B 会话 |
| AI 工程师 | 1 人 | A 墨迹分析+B Prompt（可复用）|

**总计 4 人**，3 个方向并行推进。

如果人不够，优先级顺序：**方向 A > 方向 B > 方向 C**

## 四、代码仓库管理

### 分支策略

```
main          ─ 稳定发布版本
├─ feat/ink-recording     ─ 墨迹录制（前端）
├─ feat/ink-storage       ─ 墨迹存储（Rust）
├─ feat/ink-playback      ─ 回放播放器
├─ feat/ink-analysis      ─ AI 标记节点
├─ feat/cowrite-ui        ─ 共笔会话 UI
├─ feat/cowrite-storage   ─ 共笔后端存储
├─ feat/cowrite-prompt    ─ 共笔 Prompt
├─ feat/garden            ─ 写作花园
```

### PR 规范

- 每个 PR 关联对应方向的前缀：`[A]` `[B]` `[C]`
- 示例：`[A] feat: add ink event recording on textarea change`
- 至少 1 人 review 后才可合入

## 六、验收标准

### Phase 1 完成标准

- [x] 用户在编辑器每按一次键，操作被录制并存储到本地
- [x] 用户可以打开某篇笔记的回放，看到写作过程的动画
- [x] 用户可以创建共笔会话，选择 AI 身份，和 AI 交替写至少 3 轮
- [x] 共笔内容可以保存，重新打开后恢复

### Phase 2 完成标准

- [x] 回放中有 AI 标注的停顿点和纠结点，用户可点击跳转
- [x] 共笔至少有 3 种 AI 身份可用，体验差异明显
- [x] 共笔中的 human/AI 文字有清晰的颜色区分
- [x] 桌面上可以看到一株简单的植物，会响应写作行为

---

## 七、需要你本人亲自做的事

1. **产品定义和优先级决策** —— 没人能替你决定做什么、不做、先做哪个
2. **Coding 最核心的部分** —— 墨迹录制和共笔交替逻辑你亲自写，保证落地质量
3. **Review 所有人的 PR** —— 你最了解架构，保证方向不走偏
4. **设计语言统一** —— 三个方向的 UI 风格、配色、动画节奏保持一致
