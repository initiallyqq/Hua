SpringNote 开发需求文档

说明：

开发时应以本文档作为功能边界、数据边界和交互规则依据。若界面设计文档与本文档冲突，功能逻辑以本文档为准，视觉表现以对应页面的界面设计文档为准。


一、项目定位

项目名称：SpringNote

项目类型：面向实习生的 AI 智能便签与日报生成工具。

核心目标：

让实习生可以随手记录每天做了什么、遇到了什么问题、接下来计划做什么。系统通过 AI 将这些碎片内容整理成结构化内容，并自动写入日报、周报、月报。

系统不应做成复杂任务管理工具。核心是：

* 快速记录
* AI 整理
* Markdown 存储
* 日报、周报、月报生成
* Markdown 编辑和预览
* FIM 编辑补全
* 时间、收益、热力图、模型用量统计


二、核心页面与模块

当前主要模块：

* 首页
* 便签页
* 回忆书页
* 设置页
* 桌面组件

首页负责随手记录和今日概览。

便签页负责日报、周报、月报的 Markdown 源码编辑和实时预览。

回忆书页负责基于历史记录的 AI 问答和回忆检索。

设置页负责用户配置、供应商配置、默认模型配置、快捷键配置和统计查看。

桌面组件负责展示工作计时、金币收益等轻量状态，并提供快速控制入口。


三、首页功能逻辑

首页输入框用于记录用户当天的零散工作信息。

用户可能输入的内容包括：

* 完成了什么
* 正在做什么
* 遇到了什么问题
* 问题如何处理
* 明天或接下来计划做什么
* 临时想法或工作备忘

用户提交后，调用“智能生成模型”进行结构化整理。

AI 生成结果至少应整理为：

* 完成事项
* 问题记录
* 明日计划

生成结果需要执行两个动作：

1. 展示到首页底部的今日工作概览区域。
2. 合并写入当天日报 Markdown 文件。

写入日报时不能简单追加。应读取当天已有日报内容，将新生成内容与已有内容交给 AI 重新整理后写回，避免重复、冲突和格式混乱。


四、日报、周报、月报生成规则

1. 日报

日报是系统的基础记录单位。

日报来源：

* 首页随手记录
* AI 结构化整理结果
* 用户在便签页中的手动编辑

日报写入规则：

* 每天对应一个 Markdown 文件
* 文件已存在时，读取原内容后合并整理
* 文件不存在时，创建新文件
* AI 写回时必须遵守日报系统提示词

2. 周报

周报在每周一生成。

生成规则：

* 读取上一周的全部日报 Markdown 文件
* 调用 AI 总结上一周日报
* 按周报系统提示词生成结构化周报
* 写入对应周报 Markdown 文件

周报内容重点：

* 本周主要工作
* 关键进展
* 遇到的问题
* 解决方案
* 风险与阻塞
* 下周计划

3. 月报

月报在每月 1 号生成。

生成规则：

* 读取上个月的全部周报 Markdown 文件
* 调用 AI 总结月度工作
* 按月报系统提示词生成结构化月报
* 写入对应月报 Markdown 文件

月报内容重点：

* 本月核心成果
* 项目进展
* 关键问题复盘
* 个人成长
* 下月计划


五、便签页功能逻辑

便签页用于查看、编辑和预览日报、周报、月报。

页面结构：

* 左侧：便签列表
* 中间：Markdown 源码编辑区
* 右侧：Markdown 渲染预览区

中间区域必须是 Markdown 源码，不是富文本编辑器。

右侧区域必须是 Markdown 渲染预览，不是源码展示。

中间编辑区和右侧预览区应保持等宽。

便签列表用于切换不同 Markdown 文件。

便签页需要支持日报、周报、月报文件的展示和编辑。通过左侧栏右上角的三个点来切换 点击三个点后会出现列表选择日报周报月报


六、Markdown 实时渲染规则

右侧 Markdown 预览区需要在编辑区内容变化后立即刷新。

触发刷新条件：

* 用户新增内容
* 用户删除内容
* 用户修改内容
* AI 补全内容被接受后写入

预览区必须支持：

* 标题
* 段落
* 引用
* 有序列表
* 无序列表
* 表格
* 行内代码
* 代码块
* 图片
* 链接
* 数学公式

预览区只负责渲染，不负责编辑。


七、FIM 编辑补全规则

便签编辑区需要调用 FIM 模型进行编辑补全。

FIM 补全目标：

根据当前光标位置的前文和后文，预测用户接下来可能输入的内容。

展示规则：

* AI 预测内容用浅色字体显示
* 预测内容不是正式内容，只有用户接受后才写入 Markdown
* 用户继续输入时，立即隐藏上一次预测内容

快捷键规则：

* Tab：接受全部预测
* Ctrl + L：接受单行预测
* Ctrl + K：接受单个字

调用规则：

* 用户停止输入 300ms 后再调用 FIM 模型
* 必须做防抖，不允许每输入一个字符都调用模型
* 用户继续输入时，取消或废弃上一次预测结果
* 光标位置变化后，旧预测结果失效
* 文档内容变化后，旧预测结果失效

实现时要区分：

* 当前真实 Markdown 内容
* 当前 AI 预测内容
* 用户已接受的预测内容

只有用户接受预测后，预测内容才进入真实 Markdown 内容。


八、回忆书功能逻辑

回忆书用于基于历史记录进行 AI 问答。

回忆书不是追加记忆入口，而是 AI 回答入口。

用户输入问题后，系统应基于历史日报、周报、月报内容进行检索和回答。

回忆书使用“回忆书模型”。

回忆书页面状态：

* 入口态
* 问答态

入口态用户点击发送按钮后进入问答态。

右上角新对话按钮用于清空当前问答并回到入口态。
这个对话记录写到remember.json AI  人 工具调用等 新增对话后清空这个json文件内容


九、桌面组件规则

桌面组件用于轻量展示工作状态、计时、金币收益等信息。

组件交互规则：

* 鼠标左键单击组件：开始或暂停当前计时状态
* 如果当前是开始状态，左键单击后暂停
* 如果当前是暂停状态，左键单击后开始
* 每天第一次启动软件时，默认状态为开始
* 鼠标右键单击组件：打开 SpringNote APP 首页

组件状态应影响统计：

* 开始状态下累计工作时间
* 暂停状态下不累计工作时间
* 金币收益只在开始状态下增长

每天启动默认开始，是“每日首次启动应用”的默认行为。若用户当天手动暂停后没有退出应用，不应自动重新开始，除非用户再次点击组件或重新进入新的自然日启动流程。


十、数据存储规则

1. Markdown 文件

日报、周报、月报正文使用 Markdown 文件存储。

每个日报、周报、月报对应一个独立 Markdown 文件。

推荐目录：

* notes/daily/
* notes/weekly/
* notes/monthly/

推荐文件名：

* 日报：YYYY-MM-DD.md
* 周报：YYYY-WW.md
* 月报：YYYY-MM.md

Markdown 文件是正文内容的主存储来源。

不要把日报、周报、月报正文作为主内容存入 SQLite。

2. SQLite

SQLite 用于统计和结构化数据。

SQLite 存储内容包括：

* 工作时间
* 金币收益
* 首页热力图数据
* 便签编辑补全次数
* 智能生成次数
* 日报数量
* 周报数量
* 月报数量
* 模型调用记录
* 输入 Tokens
* 输出 Tokens
* 缓存 Tokens
* 应用启动次数

3. config.json

config.json 用于存储用户可修改配置。

config.json 存储内容包括：

* 每日工作时长
* 日薪
* 所在行业
* 应用字体
* 字体大小
* 开机自启动
* 显示更新
* 显示桌面组件
* 供应商配置
* 默认模型配置
* 快捷键配置


十一、模型配置规则

默认模型只保留三个业务模型：

* 智能生成模型
* 编辑补全模型
* 回忆书模型

智能生成模型用于：

* 首页随手记录结构化
* 日报合并整理
* 周报生成
* 月报生成

编辑补全模型用于：

* 便签页 FIM 补全

回忆书模型用于：

* 回忆书问答
* 历史记录检索回答


十二、设置页规则

设置页包含：

* 偏好设置
* 供应商
* 默认模型
* 快捷键
* 统计
* 关于

偏好设置包含：

* 个人信息
* 字体与显示
* 行为与启动
* 组件设置

组件设置目前只包含：

* 显示桌面组件

快捷键页面目前只保留：

* 显示/隐藏页面

统计页面需要展示：

* 总结数
* 编辑补全次数
* 总记录数
* 日报数
* 周报数
* 月报数
* 输入 Tokens
* 输出 Tokens
* 缓存 Tokens
* 应用启动次数
* 聊天热力图
* 模型用量趋势


十三、开发注意事项

* 所有 AI 生成内容都应允许用户继续编辑
* Markdown 文件必须可直接被普通文本编辑器打开
* AI 写入日报时要避免重复内容
* FIM 补全不能打断用户输入
* 统计数据和正文内容必须分开存储
* 用户配置和统计数据必须分开存储
* 模型供应商配置变化后，应影响后续调用，不应破坏已有 Markdown 内容
* UI 原型只表达交互和视觉方向，真实开发时以本文档约束功能逻辑
